Por que quatro pastas em vez de uma?

Seu projeto segue um estilo chamado Clean Architecture (arquitetura em camadas). A ideia central é simples: o núcleo do sistema não pode depender de detalhes técnicos. O conceito de "Campanha" deveria existir mesmo que você trocasse o SQL Server por outro banco, ou o ASP.NET Core por outro framework web.

Por isso o código foi separado em quatro projetos (.csproj), cada um com uma responsabilidade e uma direção de dependência — tudo aponta para dentro, para o Domain:

O ApiCampaignDash.Domain.csproj é o único projeto sem nenhum pacote NuGet — nem EF Core, nem AutoMapper. É C# puro.

Camada 1 · Domain

Entidades: o molde de uma linha de tabela

ApiCampaignDash.Domain/Entities/

Uma Entity (entidade) é uma classe C# comum cujas propriedades espelham as colunas de uma tabela. Pense nela como um molde de formulário em branco: toda vez que o Entity Framework lê uma linha do banco, ele preenche uma cópia desse molde com os valores daquela linha.

Analogia

Se o banco de dados é um arquivo de fichas físicas, a Entity é o formato impresso da ficha (os campos que ela tem). O DbContext é o bibliotecário que sabe em qual gaveta cada tipo de ficha fica guardado.

Veja Campaign.cs, a entidade central do projeto — representa uma campanha de televendas:

ApiCampaignDash.Domain/Entities/Campaign.cs
using System.ComponentModel.DataAnnotations;

namespace ApiCampaignDash.Domain.Entities
{
    public class Campaign
    {
        [Key]
        public int IdCampaign { get; set; }

        public DateTime? CompetenceDate { get; set; }

        [Required(ErrorMessage = "O campo Situação do Período... é obrigatório.")]
        public int IdCompetencePeriodStatus { get; set; }

        [Required]
        [StringLength(150)]
        public string Description { get; set; } = string.Empty;

        public int IdAssessmentType { get; set; }
        public int IdCalculationMethod { get; set; }
        public bool? ConsidersExclusives { get; set; }
        // ... demais campos (datas, ranking, observações)
    }
}

Repare em três coisas didaticamente importantes:

  • [Key] diz ao Entity Framework "esta propriedade é a chave primária da tabela" — sem isso, o EF não sabe identificar uma linha unicamente.
  • [Required] e [StringLength] são Data Annotations: validações declarativas que também acabam virando restrições de coluna no banco (NOT NULL, tamanho máximo).
  • Não existe nenhuma propriedade como public Campaign CampanhaRelacionada { get; set; } — ou seja, esta entidade não tem "navigation properties". As relações entre Campanha, Cliente e Produto são feitas só por números de ID (IdCampaign), sem o EF montar objetos relacionados automaticamente. É uma escolha válida quando o banco é legado e você quer manter o mapeamento simples e explícito.

Chaves compostas: quando um ID sozinho não basta

Clients.cs e Product.cs representam a associação entre uma campanha e seus clientes/produtos elegíveis — e usam duas propriedades marcadas com [Key]:

ApiCampaignDash.Domain/Entities/Clients.cs
public class Clients
{
    [Key] public int IdCampaign { get; set; }
    [Key] public int IdClients { get; set; }

    public string? ClientName { get; set; } = string.Empty;
    public string? IsValid { get; set; } = string.Empty;
}

Isso significa: "a linha só é única pela combinação de campanha + cliente" — exatamente como uma tabela associativa N:N em SQL. A chave composta em si é configurada na camada Infrastructure, que você vê a seguir.

Enums: o vocabulário fechado do domínio

ApiCampaignDash.Domain/Enums/

No commit b00a6e5 você criou quatro enums para dar nome aos números "mágicos" que já existiam nas colunas int (tipo de apuração, tipo de cálculo, tipo de meta, tipo de pessoa). Um enum é basicamente uma lista fechada de opções válidas, com nome, em vez de um número solto que só faz sentido se você souber decorar a tabela de domínio:

ApiCampaignDash.Domain/Enums/PersonType.cs
public enum PersonType
{
    Gerente = 1,
    Supervisor = 2,
    Vendedor = 3,
    GrandesContas = 4,
    PessoaInterna = 5,
    Televendas = 6
}

Combinado com o JsonStringEnumConverter registrado no Program.cs (você vai ver isso na camada API), o cliente da sua API recebe "Supervisor" no JSON em vez de 2 — muito mais legível sem precisar de tabela de referência.

Curiosidade do seu código

Você tem dois artefatos com o mesmo nome: a Entity ApiCampaignDash.Domain.Entities.PersonType (mapeada para uma tabela de domínio no banco) e o Enum ApiCampaignDash.Domain.Enums.PersonType (usado nos DTOs). Isso é intencional e funciona, mas exige atenção ao escrever using — o compilador vai reclamar de ambiguidade se os dois namespaces estiverem abertos no mesmo arquivo.

Interfaces de repositório: o contrato antes da implementação

ApiCampaignDash.Domain/Interfaces/

O Domain também define o que um repositório deve saber fazer, sem dizer como. É um contrato genérico...

ApiCampaignDash.Domain/Interfaces/IBaseRepository.cs
public interface IBaseRepository<T> where T : class
{
    Task<IEnumerable<T>> GetAllAsync();
    Task<T?> GetByIdAsync(int id);
}

...e uma versão específica que estende o contrato genérico com uma pergunta que só faz sentido para Campanhas:

ApiCampaignDash.Domain/Interfaces/ICampaignRepository.cs
public interface ICampaignRepository : IBaseRepository<Campaign>
{
    Task<IEnumerable<Campaign>> GetByPeriodCampaignAsync(DateTime datetime);
}

Por que isso mora no Domain e não no Infrastructure (que é quem realmente implementa)? Porque assim o Domain nunca precisa saber que existe Entity Framework. Ele só diz "alguém, em algum lugar, sabe buscar campanhas por período" — quem cumpre essa promessa é decidido depois.

Camada 2 · Infrastructure

DbContext, DbSets e Configurations: a ponte para o SQL Server

ApiCampaignDash.Infrastructure/Data/

O AppDbContext é a peça mais importante desta camada. Ele é a sessão de conversa com o banco: cada DbSet<T> declarado nele é uma "gaveta" que representa uma tabela inteira, pronta para ser consultada com LINQ.

ApiCampaignDash.Infrastructure/Data/AppDbContext.cs
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Campaign> Campaigns { get; set; }
    public DbSet<Clients> Clients { get; set; }
    public DbSet<Product> Products { get; set; }
    public DbSet<CampaignResult> CampaignResults { get; set; }
    // ... 14 outros DbSets (AwardList, ProductLine, Manufacturer, etc.)

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
    }
}

Analogia

Se cada DbSet<Campaign> fosse uma frase, seria: "AppDbContext, me dá acesso à tabela de Campanhas como se fosse uma lista de objetos Campaign em memória." Você escreve _context.Campaigns.Where(...) e o EF Core traduz isso para SELECT ... FROM tblCampanhaTelevendas WHERE ... por trás dos panos.

Repare que o OnModelCreating não configura nada diretamente — ele delega para ApplyConfigurationsFromAssembly, que varre o projeto Infrastructure procurando por qualquer classe que implemente IEntityTypeConfiguration<T>. É aqui que mora a parte mais interessante do seu mapeamento:

Configurations: onde a "tradução" entre inglês e o banco legado acontece

ApiCampaignDash.Infrastructure/Data/Configurations/

Suas entidades têm nomes em inglês (Description, StartDate), mas o banco GS300GP é legado e usa nomes em português (DescricaoCampanha, DataInicioApuracao). Isso é resolvido com Fluent API — configuração via código, em vez de atributos na classe:

ApiCampaignDash.Infrastructure/Data/Configurations/CampaignConfig.cs
public class CampaignConfig : IEntityTypeConfiguration<Campaign>
{
    public void Configure(EntityTypeBuilder<Campaign> builder)
    {
        builder.ToTable("tblCampanhaTelevendas");
        builder.HasKey(x => x.IdCampaign);

        builder.Property(x => x.IdCampaign)
               .HasColumnName("IDCampanhaTelevendas");
        builder.Property(x => x.CompetenceDate)
               .HasColumnName("DataCompetencia");
        builder.Property(x => x.Description)
               .HasColumnName("DescricaoCampanha")
               .HasMaxLength(150);
        // ... demais colunas
    }
}

E para Clients/Product, a configuração também resolve dois problemas ao mesmo tempo: a chave composta e um campo que não existe de verdade na tabela (vem de um JOIN feito na hora da consulta):

ApiCampaignDash.Infrastructure/Data/Configurations/ClientsConfig.cs
public class ClientsConfig : IEntityTypeConfiguration<Clients>
{
    public void Configure(EntityTypeBuilder<Clients> builder)
    {
        builder.ToTable("tblCampanhaTelevendasCliente");
        builder.HasKey(x => new { x.IdCampaign, x.IdClients }); // chave composta

        builder.Property(x => x.IdClients).HasColumnName("IDCliente");
        builder.Ignore(x => x.ClientName); // não é coluna, vem de JOIN em runtime
    }
}

builder.Ignore(...) é o jeito de dizer ao EF Core: "esta propriedade existe na classe C#, mas não tente gravar ou ler ela como coluna — ela é preenchida manualmente pelo repositório."

Repositórios: quem de fato conversa com o DbSet

ApiCampaignDash.Infrastructure/Data/Repositories/

Seu projeto mistura dois estilos de repositório, e isso é uma decisão de design legítima, não uma inconsistência:

Estilo 1 — Repositório genérico, para CRUD simples

ApiCampaignDash.Infrastructure/Data/Repositories/BaseRepository.cs
public class BaseRepository<T> : IBaseRepository<T> where T : class
{
    protected readonly AppDbContext _context;
    protected readonly DbSet<T> _dbSet;

    public BaseRepository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public async Task<IEnumerable<T>> GetAllAsync()
        => await _dbSet.AsNoTracking().ToListAsync();

    public async Task<T?> GetByIdAsync(int id)
        => await _dbSet.FindAsync(id);
}

CampaignRepository herda dessa classe e só adiciona o que é específico de Campanha:

ApiCampaignDash.Infrastructure/Data/Repositories/CampaignRepository.cs
public class CampaignRepository : BaseRepository<Campaign>, ICampaignRepository
{
    public CampaignRepository(AppDbContext context) : base(context) { }

    public async Task<IEnumerable<Campaign>> GetByPeriodCampaignAsync(DateTime datetime)
        => await _context.Campaigns
                 .AsNoTracking()
                 .Where(c => c.CompetenceDate == datetime)
                 .ToListAsync();
}

Estilo 2 — Repositório "avulso", para consultas de relatório com SQL bruto

Quando a consulta é um relatório complexo com JOINs e CASE WHEN, forçar isso dentro do padrão genérico não ajudaria em nada — então o repositório implementa a interface direto, usando Database.SqlQuery<T>:

ApiCampaignDash.Infrastructure/Data/Repositories/CampaignResultReportRepository.cs
public async Task<IEnumerable<CampaignResultReport>> GetResultsAsync(DateTime competenceDateFrom)
{
    var results = await _context.Database.SqlQuery<CampaignResultReport>($"""
        SELECT tblCamTelRes.IDCampanhaTelevendas AS IdCampaign,
               tblCamTelRes.DescricaoCampanha AS CampaignDescription,
               CASE WHEN tblCamTel.IDCampanhaTelevendasTipoApuracao = 1 THEN 'POSITIVAÇÃO'
                    WHEN tblCamTel.IDCampanhaTelevendasTipoApuracao = 2 THEN 'VENDAS' END AS CampaignTypeDescription
               -- ... demais colunas
        FROM GS300GP.dbo.tblCampanhaTelevendas tblCamTel (NOLOCK)
        LEFT JOIN GS300GP.dbo.tblCampanhaTelevendasResultado tblCamTelRes (NOLOCK)
               ON tblCamTelRes.IDCampanhaTelevendas = tblCamTel.IDCampanhaTelevendas
        WHERE CAST(tblCamTel.DataCompetencia AS DATE) >= {competenceDateFrom}
        """).ToListAsync();

    return results;
}

Por que isso é seguro

Mesmo usando uma string interpolada ($"""..."""), o competenceDateFrom dentro de {...} não vira concatenação de texto — o EF Core transforma isso em um parâmetro SQL de verdade (@p0). É por isso que dá pra usar interpolação aqui sem abrir brecha para SQL Injection, diferente de montar a string você mesmo com +.

Camada 3 · Application

Services e DTOs: as regras de negócio e o "envelope" de saída

ApiCampaignDash.Application/

Essa camada existe para responder: o que o sistema faz com as entidades, e o que exatamente ele expõe para fora. Duas peças fazem esse trabalho: Services e DTOs.

DTO: o envelope que você entrega, não o objeto interno

ApiCampaignDash.Application/DTOs/

Um DTO (Data Transfer Object) parece uma entidade, mas tem um papel diferente: é o formato exato que sai pela API. Isso permite, por exemplo, transformar um int cru em um enum tipado sem alterar a entidade original:

ApiCampaignDash.Application/DTOs/CampaignDto.cs
public class CampaignDto
{
    public int IdCampaign { get; set; }
    public string Description { get; set; } = string.Empty;

    // na Entity isto é "int IdAssessmentType" — aqui já chega tipado:
    public AssessmentType? IdAssessmentType { get; set; }
    public CalculationMethod? IdCalculationMethod { get; set; }
    // ... demais campos espelhando Campaign
}

Analogia

Se a Entity é a ficha interna do arquivo (com toda a informação bruta de gestão interna), o DTO é o formulário que você entrega para um visitante: só o que ele precisa ver, no formato que faz sentido pra ele — não necessariamente idêntico ao original.

AutoMapper: quem copia Entity → DTO automaticamente

ApiCampaignDash.Application/Mappings/

Copiar campo por campo de Campaign para CampaignDto na mão seria repetitivo. O AutoMapper resolve isso com um "perfil" que declara a correspondência uma única vez:

ApiCampaignDash.Application/Mappings/CampaignProfile.cs
public class CampaignProfile : Profile
{
    public CampaignProfile()
    {
        CreateMap<Campaign, CampaignDto>();
    }
}

Como os nomes das propriedades batem (Description → Description), o AutoMapper copia sozinho — inclusive convertendo int em enum por convenção, quando os valores numéricos coincidem.

Service: quem orquestra Repository + Mapper

ApiCampaignDash.Application/Services/

O Service é a camada que o Controller efetivamente chama. Ele não sabe nada sobre SQL Server nem sobre HTTP — só sabe pedir dados ao repositório e devolver DTOs prontos:

ApiCampaignDash.Application/Services/CampaignService.cs
public class CampaignService : ICampaignService
{
    private readonly ICampaignRepository _repository;
    private readonly IMapper _mapper;

    public CampaignService(ICampaignRepository repository, IMapper mapper)
    {
        _repository = repository;
        _mapper = mapper;
    }

    public async Task<IEnumerable<CampaignDto>> GetAllAsync()
    {
        var campaigns = await _repository.GetAllAsync();
        return _mapper.Map<IEnumerable<CampaignDto>>(campaigns);
    }

    public async Task<CampaignDto?> GetByIdAsync(int id)
    {
        var campaign = await _repository.GetByIdAsync(id);
        return campaign is null ? null : _mapper.Map<CampaignDto>(campaign);
    }
}

Note a injeção de dependência no construtor: o CampaignService recebe ICampaignRepository (a interface do Domain), não CampaignRepository (a implementação concreta do Infrastructure). Isso é o que permite ao Program.cs decidir, num único lugar, qual implementação entregar — e permite trocar a implementação nos testes sem tocar no Service.

Camada 4 · API

Controllers e Program.cs: a porta de entrada HTTP

ApiCampaignDash/

Esta é a única camada que sabe o que é uma requisição HTTP. Um Controller é o recepcionista: recebe o pedido, chama quem realmente resolve (o Service) e devolve uma resposta HTTP formatada.

ApiCampaignDash/Controllers/CampaingController.cs
[Route("api/[controller]")]
[ApiController]
public class CampaignController : ControllerBase
{
    private readonly ICampaignService _campaignService;

    public CampaignController(ICampaignService campaignService)
        => _campaignService = campaignService;

    [HttpGet]
    public async Task<ActionResult<IEnumerable<CampaignDto>>> GetAll()
    {
        var campaigns = await _campaignService.GetAllAsync();
        return Ok(campaigns);
    }

    [HttpGet("{id:int}")] // GET api/campaign/5
    public async Task<ActionResult<CampaignDto>> GetById(int id)
    {
        var campaign = await _campaignService.GetByIdAsync(id);
        return campaign is null ? NotFound() : Ok(campaign);
    }
}

[Route("api/[controller]")] vira api/Campaign automaticamente (o token [controller] é substituído pelo nome da classe sem o sufixo Controller). O arquivo se chama CampaingController.cs (com um erro de digitação no nome do arquivo — "Campaing" em vez de "Campaign"), mas a classe dentro dele está correta como CampaignController.

Program.cs: onde tudo é ligado

É aqui que as peças das quatro camadas são registradas no container de injeção de dependência — o mecanismo que decide, em tempo de execução, qual classe concreta entregar quando alguém pede uma interface:

ApiCampaignDash/Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));

builder.Services.AddAutoMapper(cfg => { }, typeof(CampaignProfile));

builder.Services.AddScoped<ICampaignRepository, CampaignRepository>();
builder.Services.AddScoped<ICampaignService, CampaignService>();
// ... um par Repository/Service por entidade (Manufacturer, Product, Clients, ...)

builder.Services.AddControllers()
    .AddJsonOptions(options =>
        options.JsonSerializerOptions.Converters.Add(new JsonStringEnumConverter()));

builder.Services.AddCors(options =>
    options.AddPolicy("AllowElectronApp", policy =>
        policy.WithOrigins("http://localhost:5173", "http://192.168.3.208:5173")
              .AllowAnyMethod().AllowAnyHeader()));

var app = builder.Build();
app.UseCors("AllowElectronApp");
app.UseAuthorization();
app.MapControllers();
app.Run();

Analogia

AddScoped<ICampaignRepository, CampaignRepository>() é como preencher uma ficha de "quando alguém pedir uma interface X, entregue uma instância de classe Y". Scoped especificamente significa: "crie uma instância nova por requisição HTTP, e reutilize essa mesma instância dentro de toda a requisição" — é por isso que o AppDbContext injetado no Repository e usado pelo Service é sempre o mesmo dentro de um único request.

Duas configurações valem atenção especial:

  • CORS restrito a uma origem específica (localhost:5173) — isso indica que o front-end é uma aplicação Vite (provavelmente Electron), e só ela tem permissão de chamar essa API a partir do navegador.
  • JsonStringEnumConverter — é o que faz os enums do Domain (AssessmentType, PersonType...) chegarem como texto legível no JSON, em vez de número.

O caminho completo de uma requisição

Juntando as quatro camadas, é assim que GET /api/Campaign/5 percorre seu código até virar uma linha do SQL Server e voltar como JSON:

Cliente GET /api/Campaign/5
→
API CampaignController
→
Application CampaignService
→
Infrastructure CampaignRepository
→
Infrastructure AppDbContext.Campaigns
→
SQL Server tblCampanhaTelevendas

E a volta faz o caminho inverso, mas trocando de "roupa" duas vezes:

SQL Server linha da tabela
→
Domain Entity Campaign
→
Application AutoMapper → CampaignDto
→
API JSON (enum como texto)

Três observações sobre o estado atual do seu projeto

Atenção — segurança

O appsettings.json versionado no Git contém usuário e senha em texto puro na connection string (ConnectionStrings:DefaultConnection). Vale mover isso para User Secrets (ambiente local), variáveis de ambiente, ou um cofre de segredos — e trocar a senha atual, já que ela ficou exposta no histórico do repositório.

Estrutura

O projeto ApiCampaignDash.UI existe na solução mas está vazio — reservado, provavelmente, para uma futura interface própria, já que hoje o consumidor real parece ser uma aplicação Electron/Vite externa (conforme a política de CORS no Program.cs).

Nomenclatura

app.UseHttpsRedirection() está comentado no Program.cs (HTTPS desabilitado), e não há nenhum esquema de autenticação registrado — hoje qualquer origem que passe pelo CORS consegue chamar a API sem se autenticar. Bom manter em mente se a API sair de uma rede local controlada.
